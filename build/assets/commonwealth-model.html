8Per-cycle multiplier μ (fee gain vs harvest leakage)

Each cycle, the vault's LP value is pushed up by its share of total fees and pushed down by the 25% leakage on the harvested 0.4% slice. Net change = μ.

ηcycle = Δfeetotal × (LPvault / T) / Vvault ← fee yield on vault value
δcycle = 0.004 × 0.25 = 0.001 ← leakage from harvest (25% of 0.4%)
μ = (1 + ηcycle) × (1 − δcycle) ← net per-cycle multiplier on vault value

—

9Time-based factorial math (over n cycles)

Vault value grows geometrically: each cycle, the prior cycle's value × μ. Cumulative outflows (bounty, burn, reward) are geometric series — they grow each cycle as the underlying vault value grows.

Vn = V0 × μn ← vault value after n cycles
Σ output (over n) = output0 × (μn − 1) / (μ − 1) ← geometric sum of growing per-cycle outputs

∑The full equation, in one line

V(t) = N(t) ∑ i=1 seedi × n(t) ∏ k=k0(i) [   (1 + ηk)   (1 − δ)   ]

Sum across every vault that exists at time t, of its initial seed grown by the product of every per-cycle multiplier from when it was created to now.

Where every variable is

V(t)  =  total commonwealth value at time t ← what we're solving for
N(t)  =  count of vaults extant at time t ← grows monotonically; users add, none removed
seedi  =  initial value provided by user i ← the dollar amount that user committed
k0(i)  =  cycle in which vault i was created ← the user's join time
n(t)  =  ⌊t / 29 days⌋ ← cycle index at time t
ηk  =  f × (Σj zapj,k + Vext,k) × (Ltotal,k / Tk) / Vtotal,k ← LP fee yield at cycle k, from ALL volume sources
δ  =  0.004 × 0.25 = 0.001 ← harvest leakage, fixed by contract
Vext,k  =  (external_buys + external_sells)k ← natural volume in cycle k
f  =  0.0017 ← PCS V2 LP fee, fixed

That single expression encodes all four moving parts:  time (the upper bound n(t) on the product),  users (the outer sum over N(t)),  amounts (each seedi term),  natural volume (the Vext,k inside ηk).
Every variable above is computed live in Steps 1–9, given the current chain state and your inputs.

The same equation as a recurrence (iterated addition)

Multiplication is repeated addition. So the closed-form V(t) above unfolds back into a step-by-step recurrence — every term builds on the prior term, the same way Fibonacci does. This is what the action-log buttons above are walking through, one click at a time.

━━━ across time (one vault, n cycles) ━━━
Vi(n) = Vi(n−1) × μn
       = Vi(n−1) + Vi(n−1) × (μn − 1) ← prior value + this cycle's gain

━━━ across users (k vaults so far) ━━━
Vtotal(k) = Vtotal(k−1) + seedk ← prior total + the new user's deposit

━━━ across pool actions (sequential buys / vault adds) ━━━
Pool(k) = Pool(k−1) + Δaction(Pool(k−1)) ← each action reads prior pool, applies its delta

━━━ cumulative outputs (running totals) ━━━
Σ_output(t) = Σ_output(t−1) + output(t) ← prior cumulative + this cycle's flow

Each line is a one-step recurrence. The closed-form expressions earlier (V0 × μn, geometric sum, etc.) are just the recurrence "unrolled" — *evaluated* across n applications. Run the action buttons above three times in a row and you're literally walking the recurrence: click 1 applies the rule to the chain baseline, click 2 applies the same rule to click 1's output, click 3 applies it to click 2's output. Each pool state in the log is a term in the sequence; each μ k in the time projection is a multiplier between adjacent terms.

"Multiplication is repeated addition." A closed-form geometric series is repeated multiplication. Repeated multiplication is a recurrence, which is a sequence, which is — in operator terms — a structured family of sums.
The whole equation is, at base, addition: each term added to the running total, layer by layer, generation by generation.

∞Growth curve · start to finish

The chart below is not interpolated and not approximated. Every point is the result of running the canonical PCS V2 AMM formulas iteratively from the live pool snapshot — each cycle's reserves are derived from the previous cycle's reserves through the formulas spelled out below. Cycle transition · Fcycle(state, params) → state'

Primitive 1 · swap (PCS V2 with 0.25% fee)
Fswap(Rin, Rout, Δin, f) = Rout · Δin · (1−f) / [ Rin + Δin · (1−f) ]
Primitive 2 · mint (Uniswap V2 LP-token formula)
Fmint(Ra, Rb, T, Δa, Δb) = min( Δa · T / Ra , Δb · T / Rb )
Primitive 3 · burn (proportional removeLiquidity)
Fburn(Ra, Rb, T, Δlp) = ( Δlp · Ra / T , Δlp · Rb / T )
Composite · zap (the vault build flow)
Fzap(Ra, Rb, T, x) : 1. bout ← Fswap(Ra, Rb, x/2, f)   # swap half
2. (Δa, Δb) ← addLiq(Ra+x/2, Rb−bout, x/2, bout)   # pair
3. Δlp ← Fmint(post-swap reserves, Δa, Δb)   # mint LP
Fcycle · the per-29-day transition
1. External volume — apply 2N alternating Fswap calls (N buys + N sells). Each call sees the reserves left by the previous call.
2. Vault harvest:
   (hB, hR) ← Fburn(RB, RR, T, LPheld · 0.004)
   BNB side  = 10%·hB bounty  + 15%·hB zap (Fzap)  + 75%·hB recompound  = 100%
   RKL side  = 40%·hR recipient  + 60%·hR recompound  = 100%
   Recompound: FaddLiq(RB, RR, T, 0.75·hB, 0.60·hR)  # RKL is limiting → tiny BNB dust
3. For each new vault: Fzap(RB, RR, T, seedBNB)  # sequential, state shifts between vaults

Fcycle is what the chart iterates — apply it n times to walk n cycles forward. Every reserve, LP supply, and price on the chart is the output of one of these formulas operating on the previous cycle's state. Nothing is interpolated; nothing is approximated by a multiplier or growth-rate shortcut.

The asymptote is structural, not a forecast. The 40% recipient slice of every harvest leaves the pool permanently — burn vaults to dEaD, reward vaults to wallets — so pool RKL drains continuously at ~0.16% of remaining R_RKL per cycle. New vaults inject BNB and net-zero RKL once the pool is past its initial thin-pool regime, which is why R_BNB explodes (top panel) while R_RKL grinds down (bottom panel) on a much slower clock.
